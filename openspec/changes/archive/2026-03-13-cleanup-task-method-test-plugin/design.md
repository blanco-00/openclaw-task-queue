# 设计文档：简化任务管理流程并测试插件

## Context

当前 OpenClaw 使用手动 Markdown 文件管理任务（TASKS.md + 多个文档），流程繁琐且容易出错。已开发的 `@openclaw-task-queue/core` 插件提供结构化的任务队列管理功能。

**当前状态:**
- 任务存储: Markdown 文件 (TASKS.md)
- 任务领取: 手动编辑文件标记
- 状态追踪: 手动更新状态
- 文档: auto-task-system.md, ai-auto-task.md, sub-agent-task-allocation.md

**约束:**
- 需要保持与现有 OpenClaw 系统的兼容性
- 现有 TASKS.md 保留作为历史记录
- 插件已在 GitHub/Gitee 发布，需要安装测试

## Goals / Non-Goals

**Goals:**
1. 用插件替换手动任务管理流程
2. 清理重叠的任务文档
3. 本地安装插件并验证一键使用
4. 确认插件可以有效简化任务管理

**Non-Goals:**
- 不修改插件源代码（已开发完成）
- 不删除现有 TASKS.md 历史记录
- 不影响现有的 cron 任务（除非需要测试）
- 不强制要求所有任务都必须用插件

## Decisions

### D1: 插件安装方式
**决策:** 使用 npm 本地安装插件
**理由:** 
- 插件已发布到 npm (@openclaw-task-queue/core)
- 本地测试验证安装流程
- 未来可以发布到 OpenClaw 插件市场

**备选方案:**
- 直接复制源码到项目 (不采用 - 无法测试发布流程)
- 使用 git submodule (不采用 - 增加复杂度)

### D2: 文档清理策略
**决策:** 合并为简短的插件使用指南
**理由:**
- 原文档描述的是工作流原理，现在用插件实现
- 保留文档位置，减少其他引用更新
- 添加插件使用示例

**备选方案:**
- 完全删除旧文档 (不采用 - 保留历史上下文)
- 移动到 archive 文件夹 (不采用 - 需要更新引用)

### D3: 测试策略
**决策:** 手动测试 + 验证工具函数
**理由:**
- 插件工具函数可以直接调用
- 快速验证核心功能
- 不需要完整的 CI/CD

**备选方案:**
- 编写自动化测试 (不采用 - 已有单元测试)
- 端到端测试 (不采用 - 复杂度高)

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 插件安装失败 | 无法测试 | 检查 npm 权限和网络 |
| 工具函数调用失败 | 无法验证 | 查看插件 README |
| 与现有流程冲突 | 用户习惯改变 | 保留两种方式可选 |

## Open Questions

1. **Q: 插件是否需要配置认证?**
   - 当前插件使用本地 JSON 文件存储，可能需要后续支持数据库

2. **Q: 如何与现有 cron 任务集成?**
   - 可能需要修改 ai-task-executor 使用插件 API

3. **Q: 文档更新后是否需要通知用户?**
   - 建议添加简短的使用说明
